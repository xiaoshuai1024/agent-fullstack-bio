# 禁止推测与无信息开发

## 描述

当需求或信息不完整时，必须主动询问用户；禁止进行推测性开发或无信息开发。

## 适用场景

- 需求描述不完整、模糊或存在歧义
- 技术方案、边界条件、错误处理方式未明确
- 验收标准、用户故事、交互流程缺失
- 任何存在信息缺口的开发或设计任务

## 必须行为

### 1. 识别信息缺口

在开始任何设计或开发前，先确认以下信息是否完整：

| 信息类别 | 必须明确的内容 |
|---------|--------------|
| **需求** | 用户要解决的问题、目标用户、期望的业务价值 |
| **范围** | 做什么、不做什么（明确边界） |
| **交互** | 用户操作流程、系统响应、错误处理 |
| **验收** | 如何算完成、验收标准是什么 |
| **技术** | 技术方案、数据结构、API 契约 |
| **边界** | 异常情况、极限值、权限边界 |

### 2. 主动询问用户

发现信息缺口时，**必须立即停止**并使用 `AskUserQuestion` 工具询问用户：

- 问题要具体、结构化
- 提供清晰的选项（如有多种可能方案）
- 说明为什么需要这些信息
- 禁止以"先做了再说"为由继续

### 3. 暂停开发的红线

出现以下任一情况时，**必须暂停**并先澄清：

- ❌ 需求只有一句话，没有用户故事或场景
- ❌ 技术方案未明确，存在多种可选实现
- ❌ 边界条件、错误处理未定义
- ❌ 验收标准缺失或模糊
- ❌ 对用户意图存在任何疑问

### 4. 询问的方式规范

使用 `AskUserQuestion` 时：

```markdown
- 用中文提问（技术标识除外）
- 问题结构化，分点列出
- 提供选项（如有多种可能）
- 说明不澄清的影响
```

## 禁止行为

- ❌ **推测用户意图**："我觉得用户可能想要..."
- ❌ **假设性开发**："先按我想的做，不行再改"
- ❌ **盲目编码**：信息不全就开始写代码
- ❌ **跳过澄清**：以"简单"、"显然"为由省略询问
- ❌ **部分实现**：只实现自己理解的部分，忽略未明确的部分

## 例外情况

仅以下场景可例外（仍需在代码/计划中记录假设）：

- 用户明确授权："你看着办"、"按你的经验来"
- 极小的、不影响核心功能的细节（如文案措辞）
- 已有明确的、可复用的先例或规范

## 加载时机

- 任何开发或设计任务开始前
- 发现需求或信息不完整时
- 用户表达模糊或有歧义的需求时
