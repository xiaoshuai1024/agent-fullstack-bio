# 项目级文档编写规范

本文档用于统一本项目内的方案文档、技术文档、稳定性文档编写方式，确保输出内容可评审、可执行、可同步到飞书。

## 适用范围

- 需求方案文档
- 技术实现方案文档
- 监控与稳定性建设文档
- 运维与应急流程文档

## 文档结构规范

### 必须包含的结构

当文档属于“方案/建设类文档”时，至少包含以下结构：

- 背景与目标
- 当前现状与风险（若当前未建设能力，必须先写风险）
- 建设范围
- 具体方案（按系统分章）
- 流程与机制（告警/应急/复盘）
- 实施计划与交付物

### 风险章节要求

- 在文档前部增加“当前无该能力时的风险与问题”章节。
- 必须覆盖：故障发现、故障定位、影响范围、协作效率、业务风险。
- 风险描述要可落地，不写空泛结论。

### 优化效果章节要求

- 在风险章节后增加“优化后的预期效果”章节。
- 从可见性、发现时效、定位效率、响应规范、复盘闭环五个维度描述。
- 预期效果应可量化或可验证。

## 内容粒度规范

### 监控类文档必须细化到“监控项定义”

对于每类监控项，至少写清楚：

- 监控项名称（建议统一英文指标名）
- 监控项含义
- 计算口径
- 统计维度
- 告警阈值建议

### 监控对象拆分要求

监控方案必须分开写，不可混写：

- 前端系统（小程序）
- 后端系统（PHP 应用）
- 基础设施与云资源
- 日志与告警触达

## 图表规范

### 必备图表

监控与稳定性方案文档至少包含：

- 系统监控架构图
- 告警/故障处理流程图

### Mermaid 兼容规范（避免飞书解析失败）

- 节点文本使用引号，如 `A["文本"]`
- 连线标签使用 `-- "标签" -->`
- 每行建议以 `;` 结束
- 避免在节点文本中使用易冲突字符（如 `@`），必要时改写文本
- 提交前先本地预览 Mermaid 语法

## 编号与标题规范

- 文档标题默认不加编号前缀（如 `1.`、`1.1`）。
- 禁止出现从 `0` 开始的章节编号。
- 若确需编号，必须从 `1` 开始并全篇统一。

## 监控方案专项约束（老平台）

针对老平台监控建设，默认遵循以下约束：

- 告警触达方式：统一使用飞书。
- 云服务器监控：优先采用阿里云免费服务（CMS、SLS 免费额度、告警中心）。
- 核心链路优先：先覆盖登录、下单、支付、回调等核心业务。

## 飞书同步规范

### 本地优先原则

- 文档先在本地 Markdown 完成评审。
- 用户确认无误后再上传飞书。

### 上传确认流程

文档完成后，必须询问用户：

- 是否上传到飞书
- 上传到个人空间还是企业空间

### 企业空间默认规则

- 当用户选择企业空间时，默认使用已确认企业空间 `space_id=7627391105121897697`（康豆产品研发）。
- 若用户明确指定其他企业空间，以用户指定为准。

### 目录规则

- 上传时必须指定目录节点（`parent-node-token`）。
- 默认不覆盖原文档，优先新建子文档。

## 落地检查清单（发布前自检）

- 是否包含“当前风险与问题”章节
- 是否包含“优化后效果”章节
- 是否前后端分章
- 是否包含架构图和流程图
- 是否给出监控项定义（名称/含义/口径/维度/阈值）
- 是否明确告警触达方式（飞书）
- 是否明确云监控方案（阿里云免费能力）
- 标题是否去编号或编号统一且不从 0 开始
- 是否已完成“是否上传飞书 + 空间选择”确认

## 推荐文件命名

- 方案文档：`<topic>-plan.md`
- 稳定性文档：`<topic>-stability-plan.md`
- 运行手册：`<topic>-runbook.md`

示例：

- `old-platform-monitoring-stability-plan.md`
