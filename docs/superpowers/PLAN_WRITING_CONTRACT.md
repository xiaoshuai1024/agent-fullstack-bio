# 方案编写契约（`writing-plans` 产出物）

本文档定义 **`docs/superpowers/plans/*.md`** 在「大规模编码前」**必须**包含的章节与质量条，与 [docs/SUPERPOWERS.md](../SUPERPOWERS.md)、[docs/dev/SSOT-TASK-GRAPH-PLAN.md](../dev/SSOT-TASK-GRAPH-PLAN.md)、[AGENT_WORKFLOW_CONSTRAINTS.md](../dev/AGENT_WORKFLOW_CONSTRAINTS.md)、[GSTACK_AND_SUPERPOWERS.md](GSTACK_AND_SUPERPOWERS.md)（需求侧 GStack skill 与计划如何衔接）配合使用。

加载 **`writing-plans`** 编写或实质性修改 plan 时，须满足本契约；任务依赖与状态以 **`docs/superpowers/tasks/<featureId>.json`** 为唯一事实源（SSOT）。

### 产品交付硬约束（MUST，与《KANGDOU_PRODUCT_DELIVERY_CONTRACT》对齐）

编写或定稿 plan 前须阅读 [`KANGDOU_PRODUCT_DELIVERY_CONTRACT.md`](./KANGDOU_PRODUCT_DELIVERY_CONTRACT.md)。方案正文须可核对：**禁止 MVP**；C 端须覆盖 **商详、营销、搜索、履约、售后、异常态** 的可发布标准；**首页（或一级）入口** 与 **租户默认开启 + 模块管理** 的约定；**数据与接口依赖**（§3、§4、§7）；定稿与上线前 **文档—实现全量功能审查** 的安排（不得仅主路径一句带过）。

### 推荐入口与 Agent 提醒（MUST）

- **用户**：在 Cursor / OpenCode 中编写或迭代 KangDou **全栈实现方案**时，**推荐**先发送 **`/plan-template`**（命令正文：[`.agents/commands/plan-template.md`](../../.agents/commands/plan-template.md)；OpenCode 镜像：[`.agents/commands/plan-template.md`](../../.agents/commands/plan-template.md)）。该流程约定 **第一轮仅讨论稿**（范围、假设、编号待确认问题、各系统拟增量、「明确不做（防膨胀）」草案等）→ **用户补充或拍板** → **第二轮**再按本契约输出可落盘 `docs/superpowers/plans/*.md` 定稿（含文首 `taskGraph`、§4.0 / §4.2 / §4.3、§7 等）。
- **Agent**：当用户表达 **写方案 / 技术方案 / 实现计划 / 做 plan / roadmap 落盘** 等意图，且**当轮对话未**显式使用 **`/plan-template`**（或等价的「请按 plan-template」并指向该命令所定义的两段式流程）时，须在**作答开头或紧随其后的短段落**中 **主动建议**用户使用 **`/plan-template`**，并用一两句话说明：**讨论稿先对齐范围**、**定稿再满足本契约**。用户已发送该命令、已明确拒绝、或已声明沿用其他经认可的方案流程时，**不必重复纠缠**。

### 方案阶段与两门禁 Skill（MUST）

大规模编码前，方案须**同时**可被 **[`architecture-review-e2e-tdd`](../../.agents/skills/architecture-review-e2e-tdd/SKILL.md)** 与 **[`ux-product-review`](../../.agents/skills/ux-product-review/SKILL.md)** 核对——不要求把对话全文贴进 plan，但须在下列章节给出**可对齐 skill 模板的结构化摘要**（见 **§4.0、§4.1、§4.2**、§6.1、**§7、§7.2**），并在验收阶段按 [AGENT_WORKFLOW_CONSTRAINTS.md](../dev/AGENT_WORKFLOW_CONSTRAINTS.md)「§5.1」再做一轮收口。

| Skill | 方案阶段落点（本契约） | 推荐触发方式 |
|--------|------------------------|--------------|
| **architecture-review-e2e-tdd** | §3 业务逻辑、§4（含 **§4.0** 新增模块、**§4.2** 列表级交互链、**§4.3** 页面结构/原型）、§7（含 **§7.2** 脚本保障逻辑）、§6.1 架构与 E2E 门禁自检 | 写/改 plan 时加载该 skill，对照其 §2（计划须含业务逻辑、交互流、E2E 用例计划）自检 |
| **ux-product-review** | §4.1 UX 自检摘要；涉及用户可见流程时对照四主轴与 rubric | 可用 **`/super-pm`** 或显式加载 skill，将 **阻断 / 强烈建议 / 可选** 精简写入 plan |

## 0. 文首 YAML 与任务图

- 文首 YAML 须包含 **`taskGraph`**，指向本 feature 的 `docs/superpowers/tasks/<featureId>.json`。
- **同一变更批次**创建或更新该 JSON；提交前执行：  
  `node scripts/verify-plan-ssot.mjs validate <path-to-json>`。

### 0.1 分支与阶段（MUST）

方案须包含 **「分支与仓库」** 小节（可放在 §0 之后、§1 之前，或并入 §8 但标题须保留以便检索），明确 **方案编写阶段** 与 **执行阶段** 的分支策略，并与 [docs/GIT_WORKFLOW.md](../GIT_WORKFLOW.md) **§〇.4–〇.5** 一致：

| 阶段 | 须写明的内容 |
|------|----------------|
| **方案编写** | 撰写/迭代本 plan 时，**主仓与各子模块**（若适用）拟**检出**的分支：通常为 `dev`；若方案与某已存在 `feature/…` 同批迭代，须写明该分支名及原因。 |
| **执行（大规模编码）** | 开始修改业务代码或子模块跟踪文件前，须在**各受影响仓库**切换到的 **`feature/<主题>-<简短标识>`**；多系统任务时 **各仓分支名须一致**（同名 `feature/*`）。 |

**执行纪律（Agent / 人）：** 进入执行、发生首次对 Git 跟踪文件的写入前，须按方案核对当前 `git branch`；若与方案声明不符，**先** `checkout` 至方案所列 `feature/*`（或按 GIT_WORKFLOW 从 `dev` 新建后再写），**禁止**在方案未声明的分支上落码却不在方案中勘误。方案与实现分支**不得长期漂移**（例如方案写「feature/foo」、实际只在 `dev` 提交且无变更说明）。

## 1. 需求溯源

- 链接 **《原始需求》SSOT** 路径（例如 GStack 定稿文档）；若无外部文件，说明需求来源并附关键摘录。
- **非目标**（明确不做）与 **变更记录**（相对 SSOT 的增补/裁剪）须可追踪。

---

## 2. 背景与目标

- 业务问题与用户/运营价值；可度量的成功标准或对账方式。

---

## 3. 业务逻辑与契约假设

- 领域对象、**状态变迁**、与后端/API 的假设（错误模型、租户、权限边界）。
- **状态穷举**：涉及状态流转的功能，须用表格列出**所有状态、合法转换、各转换的前置条件与后置效果**。
- 影响多端的契约须写明版本或兼容策略（若适用）。

---

## 4. 交互与界面设计摘要

- **主路径**：用户操作 → 系统响应（分步列表）。
- **入口表**：菜单 / 路由 / 页面 / 关键 API（与角色、租户相关时注明）。
- **加载 / 空 / 错态**要点。
- **UX 组件选型**：表单/筛选器中的集合选择项（状态、类型、角色等）须注明用 Select/Dropdown 而非 Input；日期/时间须注明用 DatePicker。
- **枚举值显示方案**：所有 status/type/state 字段在 UI 上的中文映射方案须写明（如用 `formatter`、`<dict-tag>`、或后端返回 `xxxLabel` 字段），禁止直接暴露原始英文值。

### 4.0 按系统的新增功能模块（MUST）

方案须包含 **「按系统的新增功能模块」** 小节（标题可自拟，须可被检索，例如含「新增模块」；不得仅散落在 §5 的「既有集成」叙述中），用 **表格** 按仓库/系统列出本特性**本批次新建**的承载物（与 §5「将触碰的现有模块」区分：§5 写复用与改造，本条写**新增**）：

| 系统 / 仓库 | 新增模块或承载物（示例粒度） | 职责简述 | 与任务图 JSON 的对应 |
|-------------|------------------------------|----------|----------------------|
| `backend/` | 新包、新 Controller、新 Flyway 版本、新枚举表 | … | 任务 id 或「见任务 N」 |
| `frontend/` | 新页面路径、新分包、新 store/composable | … | … |
| `operation-backend/` | 新菜单/路由、新 View、新 API 封装 | … | … |

某仓库**本特性完全不涉及**时，表中须有一行 **「本特性不涉及」** 并写理由；**禁止**整表缺失或仅写「见代码」。

### 4.1 方案阶段 UX 自检（MUST）

在 plan 中增加一节 **「UX 自检摘要」**，按 [`.agents/skills/ux-product-review/SKILL.md`](../../.agents/skills/ux-product-review/SKILL.md) 的输出结构给出精简结果：

- **阻断**（若有则须先解决再大规模编码）
- **强烈建议**
- **可选**

若本特性涉及 **运营后台**（`operation-backend/` 或运营壳内路由／菜单），摘要中须**显式**体现已对照 **[`docs/dev/operation-backend-dev-standards.md`](../dev/operation-backend-dev-standards.md)**（列表主文案、筛选「选／输即查」、审计展示等）；完整逐条对照可在 **`/super-pm`** 全量 rubric 的 **「运营后台交互规范对照」** 节展开，plan 内至少汇总不符合 MUST 的项及拟处理方式。

不要求粘贴全文 skill，但须体现已对照 **§「审查维度」与 rubric 标题**（摘要、阻断、强烈建议、可选、需求对照等，可按特性删减）做过方案阶段审视；**阻断**项若有，须说明拟处理方式或为何可延后（默认：阻断未处理则不进入大规模实现）。

### 4.2 列表级交互链路（MUST）

在 **§4** 文首「主路径」叙述之外，须对本特性涉及的 **每个关键列表 / 表格页**（运营后台、商家端、小程序/C 端若存在列表）分别给出 **列表级交互链**：从「进入该页」到「完成本特性相关目标」止，用 **有序列表或小表** 逐步写清 **用户操作 → 预期 UI 反馈 → 触发的读/写 API 或领域状态变化**（一步一响应，不得合并为单句口号）。

若某端**无数列表页**（仅单页表单等），须显式写 **「无数列表页」**，并给出该端 **唯一主界面** 的同等粒度交互链。

### 4.3 页面结构展示与高保真原型（MUST）

凡本批次 **新增或实质改动** 的 **用户端 / 商家端 / 运营后台** 页面，定稿须在 **§4** 内对 **每一受影响页面** 给出 **页面结构展示**，使实现方 **不依赖 Figma** 也能对齐版式与交互密度。须写清：主要区块与从上到下的信息顺序、列表/表单的关键列或字段、操作按钮/链接位置、空态与错态出现位置、与枚举/字典展示方案（§4 文首）的衔接。**可用** Markdown 标题层级、列表、表格、简易 ASCII 线框等；**不得**仅用「按现有风格」一句替代。

若在方案阶段判定 **仅靠上述结构仍不足以约束** 复杂布局、多状态叠加或视觉层次，须在 **大规模编码前** 增加 **高保真原型**（与目标技术栈一致或可低成本迁入的 **可运行** 稿：临时路由、隔离页、或与正式实现共享组件库的演示页等）。须在本小节或「实现 / 执行说明」中写明：**原型如何访问**（路径/路由/启动方式）、**与正式页面的合并或下线条件**、以及 **不将 `pages/e2e/*` 专测页作为长期原型宿主**（与 `docs/dev/e2e-testing-agent-guide.md` §1.1、`AGENTS.md` 一致）。

**本特性无前端变更** 时，须显式写 **「无前端页面」** 及理由（例如产品 SSOT 声明纯 API），本小节不适用。

### 6.1 方案阶段架构 / E2E 自检（MUST）

在 plan 中增加一节 **「架构与 E2E 门禁自检」**，按 [`.agents/skills/architecture-review-e2e-tdd/SKILL.md`](../../.agents/skills/architecture-review-e2e-tdd/SKILL.md) 核对至少：

- 关键行为是否可由 **E2E 或稳定单测** 触发（避免脆弱选择器、无意义 sleep）。
- **用户旅程 → 拟映射的 E2E** 是否覆盖主路径与 P0 分支。
- 若涉及运营读模型 / 脱敏：是否与架构审查 skill 中的 **脱敏回灌** 等红线对齐（适用则勾选已考虑项）。

## 7. E2E 测试计划

- **用户旅程** → 拟新增或修改的 spec 路径、`describe`/`it` 标题（可表格）。
- 区分 **P0（合并前必绿）** 与 P1+。
- 数据、fixture、环境依赖（含 skip 约定）须写明。
- **E2E 量化表**：按功能类型标注最少用例数（见 [`.agents/rules/kangdou-testing-coverage.md`](../../.agents/rules/kangdou-testing-coverage.md)「E2E 覆盖最低标准」），并逐项确认在计划中已覆盖。

### 7.1 跨端 E2E 主路径（MUST，高优先级）

凡特性涉及 **多端或多系统**（例如 **微信小程序 → 后端 `8080` → 运营后台 `5174`**，或其它跨子仓库链路），本节须增加 **「E2E 主路径」** 小节（可表格），每条路径至少包含：

| 列 | 必填内容 |
|----|-----------|
| **路径名** | 简短名称（如「P0 交易闭环」「运营审核回写小程序」） |
| **入口端** | 小程序页面 / 运营路由 / API 起点 |
| **依赖服务** | 如 `8080` 后端、`5174` Vite、微信 CLI、`local` profile、关键 Flyway 版本等 |
| **自动化命令** | 与 [`docs/dev/e2e-testing-agent-guide.md`](../dev/e2e-testing-agent-guide.md) 一致的**完整**命令（或明确写「尚无脚本，合入前须补」） |
| **P0 标记** | 是否合并前必绿 |

**禁止**仅用「会写 E2E」等笼统句替代可执行路径表。执行与合入门禁须遵守 [`.agents/rules/kangdou-e2e-execution-contract.md`](../../.agents/rules/kangdou-e2e-execution-contract.md) 与 `e2e-testing-agent-guide.md` **§2.5**（禁止假绿、禁止降级已约定脚本、执行中测试冻结等）。

**小程序单端路径（MUST）**：§7.1 表中「入口端」须为 **产品 `pages/**` 路由**；若仍依赖 `pages/e2e/*` 专测页，须在表中**单列一行**标为 **技术债迁出中** 并指向专项计划中的迁出里程碑，**禁止**将专测页作为新增需求的默认写法。

**含 UI 的特性（MUST）**：默认验收物为 **用户/商家/运营在界面上可完成的完整链路**；计划 §7 中 E2E 不得仅用后端 IT 或 REST 冒烟替代已承诺的 §4.2 交互链。

### 7.2 交互链路保障脚本逻辑（MUST）

对 **§7.1** 表中**每条**路径（以及 plan 中声明的 **P0 单端专用脚本**），须写清 **保障逻辑**（可在 §7.1 表格中**加列**扩写，或在本小节按路径逐条列出），至少包含：

| 要素 | 须写明的内容 |
|------|----------------|
| **覆盖的交互链** | 对应 **§4.2** 哪条列表链 / 主路径步骤编号或标题 |
| **前置数据与 fixture** | 谁创建、是否可重复跑、清理或隔离策略 |
| **关键断言** | 每步验证什么（HTTP、DB、UI 文案、禁止裸枚举英文、权限边界等）；何处 **失败即停** 与仓库 bail 约定一致 |
| **合并门禁** | 是否 P0 必绿；是否允许 skip 及条件（与 `e2e-testing-agent-guide.md` 一致） |

若脚本 **尚未落地**，「自动化命令」列仍须按 §7.1 写 **「尚无，合入前须补」**，且 **本条不得省略**：须写 **拟** 保障逻辑（预期断言轮廓与数据依赖），禁止整行空白或仅写「后续补充」而无断言级说明。
